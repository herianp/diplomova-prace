# Plan 25-01: Onboarding Completion Button & State Fix

## Goal
Add a "Complete" button to the onboarding wizard (Step 3, browse mode) that appears after sending at least one join request, and fix onboarding page reappearing on refresh by properly setting `onboardingCompleted` in Firestore.

## Analysis

**Root cause of both issues:** After sending join requests, the user has no way to mark onboarding as complete without either creating a team or getting approved. On page refresh, `onboardingCompleted` is false in Firestore, so the route guard redirects back to `/onboarding`.

**Solution:** The "Complete" button calls the existing `completeOnboarding()` (sets `onboardingCompleted=true` in Firestore + store), then navigates to `/teams`. This fixes both the UX issue and the refresh issue.

## Changes

### 1. `src/components/onboarding/TeamBrowseList.vue`
- Emit `has-pending-requests` event (boolean) whenever `userRequests` changes
- Use `watch` on `userRequests` to emit `true` when any pending requests exist, `false` otherwise

### 2. `src/composable/useOnboardingComposable.ts`
- Add `hasPendingRequests` ref (boolean, default false)
- Add `onPendingRequestsChange(hasPending: boolean)` handler to update the ref
- Add `completeAfterRequests()` function that calls `completeOnboarding()` + navigates to `/teams`
- Export all three new items

### 3. `src/components/onboarding/OnboardingWizard.vue`
- Import and use `hasPendingRequests`, `onPendingRequestsChange`, `completeAfterRequests` from composable
- Pass `@has-pending-requests="onPendingRequestsChange"` to `TeamBrowseList`
- Add "Complete" `q-btn` (color="primary") below TeamBrowseList, shown when `hasPendingRequests` is true

### 4. `src/components/onboarding/OnboardingWizardMobile.vue`
- Same changes as OnboardingWizard.vue (mirror the Complete button and event handling)

### 5. i18n keys
- Add `onboarding.complete` key in both CS and EN locale files for the button label
