  **useTeamMemberUtils.ts — Handle deleted user display centrally:**
  - Update `getMemberDisplayName` to check `member.status`:
    ```typescript
    const getMemberDisplayName = (member: ITeamMember): string => {
      const baseName = member.displayName || member.email || `Member ${member.uid.substring(0, 8)}...`
      if (member.status === 'deleted') {
        return `${baseName} (deleted)`
      }
      return baseName
    }
    ```
  - This single change propagates to ALL places that use `getMemberDisplayName` and `getPlayerName` (PlayersComponent, CashboxComponent, ReportsComponent, and any other consumer) — no need to update each component individually.
  - The "(deleted)" text is hardcoded English here. This is acceptable since display names themselves are not translated. The visual treatment (gray color) is handled in the UI layer (Plan 02 AdminUsersTab already handles this for the admin view; for other views, the suffix provides the signal).

  **useAuthUseCases.ts — Add safety net for soft-deleted users:**
  - Find the auth state change listener (onAuthStateChanged callback or equivalent).
  - After successfully getting the authenticated user, add a check: read the user's Firestore document and check if `status === 'deleted'`.
  - If status is 'deleted': sign the user out immediately (`await signOut(auth)`) and do NOT proceed with setting up team/notification listeners.
  - This handles the edge case from RESEARCH.md Pitfall 4: if Auth deletion failed but Firestore was soft-deleted, the user can still authenticate but should be blocked.
  - Add a log warning: `log.warn('Soft-deleted user attempted login', { uid })`
  - IMPORTANT: Read the user doc BEFORE setting up team listeners. The existing flow likely reads the user doc already — extend that check. Do NOT add an extra Firestore read if the user doc is already being read in the auth flow.
  - If the user doc does not have a `status` field at all (existing users before this feature), treat as active (this is the default for all current users).

  Update the `users/{userId}` rules section to allow app admins to update user documents (needed for the Cloud Function fallback and potential direct admin operations):

  Current rules:
  ```
  match /users/{userId} {
    allow read, write: if request.auth != null && request.auth.uid == userId;
    allow read: if request.auth != null;
  }
  ```

  Updated rules:
  ```
  match /users/{userId} {
    allow read, write: if request.auth != null && request.auth.uid == userId;
    allow read: if request.auth != null;
    // Allow app admin to update user documents (soft-delete)
    allow update: if isAppAdmin();
  }
  ```

  NOTE: The Cloud Function uses Admin SDK which bypasses security rules entirely. This rule is a safety net for any client-side admin operations and for consistency. The `isAppAdmin()` helper already exists in the rules file.

  Do NOT modify any other rules. Keep all existing rules exactly as-is.
